# ADR 0002 — UUID v7 em todas as PKs

**Status:** Aceito
**Data:** 2025-01

---

## Contexto

O sistema é multi-tenant e expõe IDs em URLs (`/budgets/{id}`). IDs sequenciais (`bigint`) expõem informações sensíveis (volume de orçamentos por empresa, ordem de criação) e dificultam migração / merge de bases.

---

## Decisão

Adotar **UUID v7** em todas as chaves primárias do sistema.

UUID v7 é **ordenável temporalmente** (os primeiros bits contêm timestamp), o que mantém boa performance em índices B-tree — próxima de bigint sequencial — sem os problemas de fragmentação do UUID v4.

---

## Consequências

### Positivas

- Não expõe contadores sensíveis em URLs;
- Permite geração no cliente (frontend pode criar o ID antes do POST);
- Facilita merge de bases / migração entre ambientes;
- Performance de índice próxima a bigint;
- Combina bem com multi-tenancy.

### Negativas

- Strings de 36 caracteres (vs 8 bytes em bigint) — overhead pequeno;
- Necessário garantir uso de biblioteca correta (Laravel 11+ já suporta nativamente, ou usar `ramsey/uuid` v4.7+).

---

## Implementação

```php
use Illuminate\Support\Str;

// Laravel 11+ (helper nativo)
$id = (string) Str::uuid7();

// Ou via Ramsey
use Ramsey\Uuid\Uuid;
$id = Uuid::uuid7()->toString();
```

Em migrations:

```php
$table->uuid('id')->primary();
```

Em models, usar trait `HasUuids` (Laravel) configurada para v7.

---

## Alternativas descartadas

- **bigint autoincrement** — expõe contadores, dificulta merge;
- **UUID v4** — fragmenta índices, performance inferior;
- **ULID** — boa opção, mas UUID v7 tem suporte mais amplo e padrão IETF.
